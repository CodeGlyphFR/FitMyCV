bump-version.sh — Versioning 4 parties

Script Bash exécuté automatiquement par GitHub Actions lors du merge sur main. Il scanne tous les commits depuis le dernier tag et incrémente la version selon les préfixes Conventional Commits.

Format de version

# Format : X.X.X.X (Major.Minor.Patch.Build)
# Exemple : 1.3.2.7

Règles de bump

Préfixe de commit Partie incrémentée Exemple Résultat
feat!: ou BREAKING CHANGE: 1er (Major) feat!: supprimer API v1 1.3.2.7 → 2.0.0.0
feat: 2e (Minor) feat: ajouter export PDF 1.3.2.7 → 1.6.0.0
fix: 3e (Patch) fix: corriger affichage mobile 1.3.2.7 → 1.6.0.0
Autre (chore:, docs:...) 4e (Build) chore: mettre à jour CI 1.3.2.7 → 1.3.2.8
Le plus haut niveau gagne — Si une PR contient à la fois un fix: et un feat:, c'est le feat: (Minor) qui s'applique. Le script retient toujours le niveau le plus élevé parmi tous les commits.

Propagation de la version

Après mise à jour de package.json, le script scanne dynamiquement tous les fichiers du projet (hors .git, node_modules, .next, dist) et remplace l'ancienne version par la nouvelle :

# Scan dynamique
EXCLUDE_DIRS=("--exclude-dir=.git" "--exclude-dir=node_modules" ...)
FILES_TO_UPDATE=$(grep -rl "${EXCLUDE_DIRS[@]}" "$OLD_VERSION" . || true)

# Remplacement dans chaque fichier trouvé
for FILE in $FILES_TO_UPDATE; do
    sed -i "s/$OLD_VERSION/$NEW_VERSION/g" "$FILE"
done
Fichiers impactés — La version est propagée dans docs/, README.md, les fichiers de configuration, et tout fichier contenant le numéro de version. Les package.json et package-lock.json sont traités séparément via Node.js pour préserver le formatage JSON.

run-data-migrations.js — Migrations de données

Exécute les scripts de migration de données stockés dans prisma/data-migrations/. Complémentaire aux migrations Prisma (schéma), ce script gère les transformations de données (renommage, nettoyage, calculs).

Table de suivi

-- Table créée automatiquement au premier lancement
CREATE TABLE IF NOT EXISTS "_data_migrations" (
  "id"         SERIAL PRIMARY KEY,
  "name"       TEXT UNIQUE NOT NULL,
  "applied_at" TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

Format d'un script de migration

// prisma/data-migrations/20240215_1430_normalize_skill_levels.js
module.exports = async (prisma) => {
  // Normaliser les niveaux de compétence en valeurs numériques
  const skills = await prisma.skill.findMany({
    where: { level: { not: null } }
  });

  for (const skill of skills) {
    const numericLevel = normalizeToNumber(skill.level);
    if (numericLevel !== null && numericLevel !== skill.level) {
      await prisma.skill.update({
        where: { id: skill.id },
        data: { level: numericLevel }
      });
    }
  }
};
Convention de nommage — Les fichiers suivent le format YYYYMMDD_HHMM_description.js. Le tri alphabétique garantit l'ordre chronologique d'exécution.
Exécution unique — Chaque migration n'est exécutée qu'une seule fois grâce à la table de suivi. En cas d'erreur, le script s'arrête immédiatement (process.exit(1)) pour éviter les effets de bord.

sync-stripe.mjs — Synchronisation Stripe (Production)

Script ESM qui synchronise les plans d'abonnement et packs de crédits entre la base de données et Stripe en production.

Usage

# Synchronisation avec sortie détaillée
node scripts/sync-stripe.mjs

# Mode silencieux (CI/CD)
node scripts/sync-stripe.mjs --quiet

Prérequis

Variable d'environnement Obligatoire Description
STRIPE_SECRET_KEY Oui Clé secrète Stripe (détecte automatiquement mode TEST vs LIVE)
DATABASE_URL Oui URL de connexion PostgreSQL

Variables d'environnement

Variable Obligatoire Description
STRIPE_SECRET_KEY Oui Clé Stripe TEST (staging)
DATABASE_URL Oui URL de la DB staging
STRIPE_LIVE_SECRET_KEY Non Clé Stripe LIVE (pour phases 1, 3, 4). Sans cette clé, seule la phase 2 s'exécute.

Le script fait correspondre les produits Stripe avec la DB via le nom du produit :

// Phase 1 : Match par nom de produit (Live → Test)
const testProductsByName = new Map(
  testProducts.data.map(p => [p.name, p])
);

// Phase 1 : Match des prix par montant + devise + type + intervalle
const exists = testPrices.data.some(tp =>
  tp.unit_amount === livePrice.unit_amount &&
  tp.currency === livePrice.currency &&
  tp.type === livePrice.type &&
  (livePrice.type !== 'recurring' ||
    tp.recurring?.interval === livePrice.recurring?.interval)
);

// Phase 2 : Match DB ↔ Stripe par nom
const dbNames = new Set([
  ...dbPlans.map(p => p.name),
  ...dbPacks.map(p => p.name)
]);

Format de sortie

Chaque template est exporté dans un fichier nommé {triggerName}--{slug}.json :

// prisma/email-templates/welcome--bienvenue-sur-fitmycv.json
{
  "name": "Bienvenue sur FitMyCV",
  "triggerName": "welcome",
  "subject": "Bienvenue sur FitMyCV.io !",
  "variables": ["userName", "loginUrl", "welcomeCredits"],
  "htmlContent": "<!DOCTYPE html>...",
  "designJson": { ... },
  "isActive": true,
  "isDefault": true
}
Slugification — Le nom du template est converti en slug (minuscules, sans accents, tirets) via la fonction interne slugify() : "Bienvenue sur FitMyCV" → "bienvenue-sur-fitmycv".

Usage
