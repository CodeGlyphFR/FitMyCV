Application des Améliorations

Le pipeline d'amélioration est orchestré par runImprovementPipeline() dans lib/features/cv-improvement/orchestrator.js. Il exécute 4 stages avec des imports dynamiques pour le chargement différé et un contrôle de concurrence via pLimit(5). Chaque stage émet des événements SSE pour la progression en temps réel.

Stage 1 : Classification des Skills

Fichier : lib/features/cv-improvement/stages/stage1-classify-skills.js

Fonction : classifySkills({ skills, signal, userId })

Modèle : gpt-4o-mini

STAGE 1 - Classification des compétences manquantes
Entrée
{
  "skills": [
    { "skill": "Kubernetes", "level": "intermediate" },
    { "skill": "Leadership", "level": "advanced" },
    { "skill": "Jira", "level": "beginner" },
    { "skill": "Scrum", "level": "intermediate" }
  ]
}
Prompt inline (pas de fichier .md)

Le prompt est intégré directement dans le fichier. Il demande à l'IA de classer chaque compétence dans une seule catégorie parmi :

  • hard_skills — Technologies, langages, domaines d'expertise
  • soft_skills — Compétences comportementales
  • tools — Logiciels et plateformes spécifiques
  • methodologies — Méthodes et frameworks de travail
Sortie
{
  "hard_skills": ["Kubernetes"],
  "soft_skills": ["Leadership"],
  "tools": ["Jira"],
  "methodologies": ["Scrum"]
}

Application des résultats (Stage 1)

Après la classification, chaque skill est ajouté au workingCv.skills :

// Pour hard_skills et tools : objet avec proficiency
workingCv.skills[category].push({ name: skillName, proficiency: level });

// Pour soft_skills et methodologies : string simple
workingCv.skills[category].push(skillName);

// Niveau normalise via normalizeToNumber()
// "beginner" -> 1, "intermediate" -> 2, "advanced" -> 3, etc.
Skip si pas de skills à ajouter

Si missingSkillsToAdd est vide ou null, le Stage 1 retourne immédiatement { success: true, data: null } sans appel IA.

Stage 2 : Prétraitement des Suggestions

Fichier : lib/features/cv-improvement/stages/stage2-preprocess.js

Fonction : preprocessSuggestions({ suggestions, experiences, projects, languages, extras, jobOffer, signal, userId })

STAGE 2 - Analyse Chain-of-Thought et routage
Données d'entrée
  • suggestions — Suggestions sélectionnées (avec userContext le cas échéant)
  • experiences — Expériences du CV (formatées : [index] Titre @ Entreprise (dates))
  • projects — Projets du CV (formatés : [index] Nom - Résumé)
  • languages — Langues du CV (formatées : [index] Nom - Niveau)
  • extras — Extras du CV (formatés : [index] Nom - Résumé)
  • jobOffer — Offre d'emploi pour le contexte
Prompts utilisés
  • prompts/preprocess-system.md
  • prompts/preprocess-user.md — Variables : counts, listes formatées, suggestions
Sortie structurée
// Retour de preprocessSuggestions()
{
  success: true,
  experienceImprovements: [
    { index: 0, improvements: [enrichedSuggestion1, ...] },
    { index: 2, improvements: [enrichedSuggestion2] }
  ],
  projectImprovements: [
    { index: 1, improvements: [enrichedSuggestion3] }
  ],
  newProjects: [
    { suggestion: enrichedSuggestion4, actionDescription: "..." }
  ],
  languageImprovements: [
    { index: 0, improvements: [enrichedSuggestion5] },
    { index: null, improvements: [enrichedSuggestion6] } // nouvelle langue
  ],
  extrasImprovements: [
    { index: null, improvements: [enrichedSuggestion7] } // nouveau extra
  ],
  unmatchedSuggestions: []
}
Actions multiples par suggestion

Une suggestion peut générer plusieurs actions ciblant des sections différentes. Par exemple, "Ajouter Kubernetes" peut cibler à la fois experience[2].skills_used et experience[2].responsibilities. L'IA retourne un tableau actions[] pour chaque classification.

Stage 3 : Améliorations Parallèles

Orchestration : Promise.all() avec pLimit(5) pour limiter à 5 appels OpenAI simultanés.

improveExperience()

stages/stage3-improve/improveExperience.js

Entrée : experience, suggestion, jobOffer, cvLanguage

Sortie : { modifications, reasoning }

Modifications possibles :

  • responsibilities.add[] — Nouvelles responsabilités
  • deliverables.add[] — Nouveaux résultats
  • skills_used.add[] — Nouvelles compétences

Application via applyExperienceModifications()

improveProject()

stages/stage3-improve/improveProject.js

Entrée : project (ou null pour création), suggestion, jobOffer, cvLanguage

Sortie : { modifications, reasoning, isNew }

Modes :

  • Modification : Améliorer un projet existant
  • Création : project: null + isNew: true

Application via applyProjectModifications()

improveLanguages()

stages/stage3-improve/improveLanguages.js

Entrée : languages[], suggestions[], jobOffer, cvLanguage

Sortie : { modifications, reasoning, hasChanges }

Types de modifications :

  • levelChanges[] — Changement de niveau
  • certificationChanges[] — Ajout de certification
  • newLanguages[] — Nouvelles langues
  • reorder[] — Réordonnancement par pertinence

Application via applyLanguageModifications()

improveExtras()

stages/stage3-improve/improveExtras.js

Entrée : extras[], suggestion, targetIndex, jobOffer, cvLanguage

Sortie : { action, modifications, newExtra, reasoning, hasChanges }

Actions possibles :

  • add — Créer un nouvel extra (certification, etc.)
  • update — Modifier un extra existant

Application via applyExtrasModifications()

Progression SSE (Stage 3)

Le Stage 3 émet des événements SSE via dbEmitter.emitCvImprovementProgress() pour chaque tâche terminée :

// Progression experiences
dbEmitter.emitCvImprovementProgress(userId, {
  taskId,
  stage: 'experiences',
  step: 'experiences',
  status: completedExperiences === totalExperiences ? 'completed' : 'running',
  current: completedExperiences,
  total: totalExperiences,
  itemType: 'experience',
});
Stage SSE Status Détail
preprocess running → completed Stage 1 & 2 terminés
classify completed Classification des skills terminée
experiences running (N/M) → completed Progression par expérience traitée
projects running (N/M) → completed Progression par projet traité
summary running → completed Stage 4 terminé
finalize completed Pipeline terminé

Stage 4 : Amélioration du Summary

Fichier : lib/features/cv-improvement/stages/stage4-summary/improveSummary.js

Fonction : improveSummary({ summary, improvedExperiences, improvedProjects, jobOffer, cvLanguage, signal, userId })

Exécution conditionnelle

Le Stage 4 ne s'exécute que si des expériences ou des projets ont été modifiés au Stage 3. La condition est : improvedExperiences.length > 0 || improvedProjects.length > 0. Si aucune modification n'a été apportée (uniquement des skills ou langues), le summary reste inchangé.

STAGE 4 - Mise à jour du summary
Entrées
  • summary — Summary actuel du CV
  • improvedExperiences — Détail des modifications apportées aux expériences
  • improvedProjects — Détail des modifications apportées aux projets
  • jobOffer — Offre d'emploi pour l'alignement vocabulaire
  • cvLanguage — Code langue du CV (converti via getLanguageName())
Prompts utilisés
  • prompts/summary-system.md — Variable : {{cvLanguage}}
  • prompts/summary-user.md — Variables : {{summaryContent}}, {{jobOfferContent}}, {{improvedExperiences}}, {{improvedProjects}}, {{cvLanguage}}
Sortie et application
// Retour de improveSummary()
{
  modifications: {
    description: "Nouveau texte de description du profil...",
    // ... autres champs du summary
  },
  reasoning: "Le summary a ete mis a jour pour refleter..."
}

// Application via applySummaryDiff()
workingCv.summary = applySummaryDiff(workingCv.summary, summaryResult.modifications);

Imports Dynamiques

Le pipeline utilise des imports dynamiques (await import()) pour éviter de charger les modules lourds au démarrage du serveur :

// lib/features/cv-improvement/orchestrator.js
async function getClassifySkills() {
  const module = await import('./stages/stage1-classify-skills.js');
  return module.classifySkills;
}

async function getPreprocessSuggestions() {
  const module = await import('./stages/stage2-preprocess.js');
  return module.preprocessSuggestions;
}

async function getImproveExperience() {
  const module = await import('./stages/stage3-improve/improveExperience.js');
  return module.improveExperience;
}

// ... idem pour getImproveProject, getImproveSummary,
// getImproveLanguages, getImproveExtras
Pourquoi des imports dynamiques ?

Chaque module de stage importe des dépendances lourdes (client OpenAI, prompt loader, schémas). Les imports dynamiques permettent de ne charger ces modules qu'au moment de leur utilisation, réduisant le temps de démarrage du serveur et la consommation mémoire au repos.

Gestion des Erreurs et Annulation

Chaque tâche du pipeline vérifie le signal d'annulation avant de lancer un appel IA :

// Verification d'annulation dans chaque tache
if (abortController.signal.aborted) throw new Error('Task cancelled');

// Les erreurs non-annulation sont capturees et n'arretent pas le pipeline
try {
  const result = await improveExperience({ ... });
  return { success: true, type: 'experience', index, data: result };
} catch (error) {
  if (error.message === 'Task cancelled') throw error; // Propagation
  return { success: false, type: 'experience', index, error }; // Continue
}
Résilience du pipeline

Si une tâche individuelle du Stage 3 échoue (erreur IA, timeout), elle retourne { success: false } mais n'arrête pas les autres tâches en cours. Seule l'annulation utilisateur (Task cancelled) propage l'erreur et arrête tout le pipeline. Le Stage 4 (summary) est également résilient : une erreur lors de la mise à jour du summary est ignorée sans bloquer le pipeline.
