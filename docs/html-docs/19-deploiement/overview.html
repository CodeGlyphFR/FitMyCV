Déploiement & Infrastructure

Architecture Docker multi-stage, pipelines CI/CD GitHub Actions sur runner auto-hébergé, variables d'environnement, health checks et configuration Next.js 16 standalone. Le déploiement couvre deux environnements : pre-prod (dev.fitmycv.io) et production (app.fitmycv.io).

Architecture Docker

Le Dockerfile utilise un build multi-stage en 4 étapes pour optimiser la taille de l'image finale et séparer les dépendances de build des dépendances de production.

Stage 1 — deps (Toutes les dépendances)

Installe l'ensemble des dépendances (build + dev) nécessaires à la compilation. Les packages natifs (python3, make, g++) sont requis pour compiler certains modules Node.js natifs.

FROM node:24-alpine AS deps
RUN apk add --no-cache libc6-compat python3 make g++
WORKDIR /app
COPY package*.json ./
COPY prisma ./prisma
ENV PUPPETEER_SKIP_DOWNLOAD=true
RUN npm ci

Stage 2 — production-deps (Dépendances de production)

Stage séparé qui installe uniquement les dépendances de production (--omit=dev). Cela permet d'obtenir un node_modules minimal pour l'image finale.

FROM node:24-alpine AS production-deps
RUN apk add --no-cache python3 make g++
WORKDIR /app
COPY package*.json ./
COPY prisma ./prisma
ENV DATABASE_URL="postgresql://dummy:dummy@localhost:5432/dummy"
ENV PUPPETEER_SKIP_DOWNLOAD=true
RUN npm ci --omit=dev
DATABASE_URL dummy — Une URL factice est fournie car Prisma valide la variable lors du postinstall. La vraie URL sera injectée au runtime via Docker Compose.

Stage 3 — builder (Compilation)

Compile l'application Next.js en mode production. Les variables NEXT_PUBLIC_* sont injectées comme ARG Docker car Next.js les intègre (bake) dans le JavaScript client au moment du build.

Build ARG Usage
NEXT_PUBLIC_RECAPTCHA_SITE_KEY Clé publique reCAPTCHA v3 pour le frontend
NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY Clé publique Stripe pour le checkout côté client
NEXT_PUBLIC_SITE_URL URL publique du site (ex: https://app.fitmycv.io)
NEXT_PUBLIC_APP_URL URL de l'application (identique à SITE_URL en production)

Stage 4 — runner (Image finale)

L'image de production embarque les outils système nécessaires au fonctionnement de l'application :

Package Rôle
chromium Navigateur headless pour Puppeteer (export PDF des CV + extraction d'URL)
font-noto, font-freefont Polices pour le rendu correct des PDF avec Chromium
graphicsmagick Conversion PDF → images lors de l'import de CV (via pdf2pic)
ghostscript Backend de rendu PDF pour GraphicsMagick
openssl Requis par Prisma pour les connexions SSL à PostgreSQL
libc6-compat Compatibilité glibc requise par certains binaires natifs Node.js sur Alpine
Puppeteer — La variable PUPPETEER_EXECUTABLE_PATH=/usr/bin/chromium-browser force Puppeteer à utiliser le Chromium système au lieu de télécharger sa propre version, ce qui réduit la taille de l'image.

L'image finale copie 5 éléments depuis les stages précédents :

# 1. Production node_modules (Prisma CLI, stripe, etc.)
COPY --from=production-deps /app/node_modules ./node_modules

# 2. Next.js standalone server
COPY --from=builder /app/.next/standalone ./
COPY --from=builder /app/.next/static ./.next/static
COPY --from=builder /app/public ./public

# 3. Prisma (schema + migrations + data-migrations)
COPY --from=builder /app/prisma ./prisma

# 4. Post-deploy scripts (migrations runner, stripe sync)
COPY --from=builder /app/scripts ./scripts

# 5. package.json (needed for npm run scripts)
COPY --from=builder /app/package.json ./package.json

Health Check

Le conteneur Docker embarque un health check natif qui interroge l'endpoint /api/health :

HEALTHCHECK --interval=30s --timeout=10s --start-period=40s --retries=3 \
  CMD node -e "fetch('http://localhost:'+(process.env.PORT||3000)+'/api/health') \
    .then(r=>{if(r.ok)process.exit(0);else process.exit(1)}) \
    .catch(()=>process.exit(1))"
Paramètre Valeur Description
interval 30s Fréquence de vérification
timeout 10s Temps max d'attente de la réponse
start-period 40s Délai de grâce au démarrage du conteneur
retries 3 Échecs consécutifs avant de marquer le conteneur unhealthy

Pipeline Pre-prod (preprod.yml)

Déclenché sur chaque Pull Request vers main (opened, synchronize, reopened, closed). Comprend 3 jobs :

Job 1 — CI (Validation du build)

Vérifie que l'application compile correctement sans Docker :

ci:
  if: github.event.action != 'closed'
  steps:
    - npm ci
    - cp .env.example .env && npx prisma generate
    - npm run build  # NODE_ENV=production
    - npm audit --audit-level=critical || true

Job 2 — Deploy Preview (Staging)

Déploie l'application sur dev.fitmycv.io (port 3001) avec une base de données clonée depuis la production :

  1. Clone DB — pg_dump fitmycv_prod → fitmycv_release (uniquement à l'ouverture de la PR)
  2. Sanitize — Supprime toutes les références Stripe Live pour éviter les opérations accidentelles sur les données de paiement réelles
  3. Docker Build — Build avec les clés publiques de staging
  4. Deploy — docker compose -p fitmycv-staging up -d
  5. Health Check — Attend jusqu'à 150s (30 tentatives × 5s) que l'application réponde
  6. Migrations — prisma migrate deploy + run-data-migrations.js
  7. Stripe Sync — Synchronise les produits et coupons depuis Stripe Live vers Stripe Test
Sanitization de la base de données — Les champs stripeCustomerId, stripeSubscriptionId et stripePriceId sont effacés ou remplacés par des préfixes staging_ pour éviter toute interaction avec l'API Stripe en mode live. 
-- Sanitization des données Stripe lors du clone staging
UPDATE "User" SET "stripeCustomerId" = NULL;
UPDATE "Subscription" SET
  "stripeCustomerId" = 'staging_' || "id",
  "stripeSubscriptionId" = NULL,
  "stripePriceId" = NULL,
  "status" = 'canceled';

Job 3 — Cleanup (Fermeture PR)

Nettoyage automatique lorsque la PR est fermée (merge ou rejet) :

cleanup:
  if: github.event.action == 'closed'
  steps:
    - docker compose -p fitmycv-staging down
    - docker rm -f fitmycv-preview
    - dropdb --if-exists fitmycv_release

1. Versioning automatique

Le script scripts/bump-version.sh analyse les messages de commit de la PR et incrémente la version selon les Conventional Commits (voir CLAUDE.md pour le détail des règles).

# Exécuté automatiquement dans le pipeline
./scripts/bump-version.sh "$COMMIT_MESSAGE"
NEW_VERSION=$(node -p "require('./package.json').version")
git add . && git commit -m "chore(release): v$NEW_VERSION [skip ci]"
git tag "v$NEW_VERSION"
git push origin main --tags
[skip ci] — Le commit de versioning utilise [skip ci] dans son message pour éviter de déclencher une boucle infinie de déploiements.

2. Synchronisation des branches

Après le versioning, les branches release et dev sont automatiquement synchronisées avec main pour que toutes les branches de travail héritent de la nouvelle version.

3. Build & Deploy

Les variables publiques sont extraites depuis le fichier .env du serveur de production et injectées comme --build-arg Docker :

docker build \
  --build-arg NEXT_PUBLIC_RECAPTCHA_SITE_KEY="$RECAPTCHA_KEY" \
  --build-arg NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY="$STRIPE_PK" \
  --build-arg NEXT_PUBLIC_SITE_URL="$APP_URL" \
  --build-arg NEXT_PUBLIC_APP_URL="$APP_URL" \
  -t fitmycv:latest .

4. Health check post-deploy

Après le déploiement, le pipeline attend que l'application soit prête avant de lancer les migrations. La boucle effectue 30 tentatives espacées de 5 secondes (150s max) :

for i in $(seq 1 30); do
  if docker exec fitmycv-prod node -e \
    "fetch('http://localhost:'+(process.env.PORT||3000)+'/api/health') \
     .then(r=>{if(r.ok)process.exit(0);else process.exit(1)}) \
     .catch(()=>process.exit(1))" 2>/dev/null; then
    echo "✅ Application is ready!"
    exit 0
  fi
  sleep 5
done

5. Migrations post-deploy

Une fois le health check passé, les migrations sont exécutées à l'intérieur du conteneur :

# Migrations de schéma Prisma
docker exec fitmycv-prod npx prisma migrate deploy

# Migrations de données (scripts JS dans prisma/data-migrations/)
docker exec fitmycv-prod node scripts/run-data-migrations.js

Configuration Next.js

Le fichier next.config.js configure Next.js 16 pour le déploiement en mode standalone :

const nextConfig = {
  // Mode standalone pour Docker (génère un serveur autonome)
  output: 'standalone',

  // Version et nom du site injectés comme variables d'env
  env: {
    NEXT_PUBLIC_APP_VERSION: version,  // Depuis package.json
    NEXT_PUBLIC_SITE_NAME: "FitMyCV.io",
  },

  // Optimisation des images (AVIF + WebP)
  images: {
    formats: ['image/avif', 'image/webp'],
    minimumCacheTTL: 60,
  },

  // Compression activée
  compress: true,

  // Next.js 16 : Turbopack est le bundler par défaut
  turbopack: {},

  // Packages serveur non bundlés (Puppeteer)
  serverExternalPackages: [
    'puppeteer', 'puppeteer-extra', 'puppeteer-extra-plugin-stealth'
  ],

  // Optimisation des imports de packages
  experimental: {
    optimizePackageImports: ['lucide-react', 'next-auth'],
  },
};
Option Valeur Description
output 'standalone' Génère un serveur Node.js autonome avec uniquement les fichiers nécessaires, optimal pour Docker
turbopack {} Active Turbopack, le bundler par défaut de Next.js 16 (remplace Webpack)
compress true Active la compression gzip/brotli des réponses
serverExternalPackages Puppeteer Exclut Puppeteer du bundling côté serveur (binaire natif)
allowedDevOrigins ['dev.fitmycv.io'] Autorise les requêtes cross-origin en développement depuis le domaine de staging
transpilePackages ['swiper'] Compile Swiper pour compatibilité avec le bundler
optimizePackageImports lucide-react, next-auth Tree-shaking optimisé pour ces librairies volumineuses

Webpack — Split Chunks (mode legacy)

La configuration Webpack est conservée pour compatibilité avec le flag --webpack mais est ignorée par Turbopack (bundler par défaut en Next.js 16). Elle définit 4 groupes de cache :

Cache Group Priorité Contenu
framework 40 React, React DOM, Scheduler, Next.js
auth 35 NextAuth.js
lib 30 Tous les autres node_modules
commons 20 Modules partagés entre 2+ chunks
